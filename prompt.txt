===============================================================================
PROJECT TYPE
===============================================================================

Build an Emby server plugin that acts as a subtitle provider and delegates subtitle searching/downloading to Bazarr instead of downloading subtitles directly inside Emby.

===============================================================================
PRIMARY GOALS
===============================================================================

Implement all of the following behavior directly in the new repository:

1. Register with Emby as a subtitle provider in the same general user-facing role as Emby's subtitle-provider plugins.
2. When Emby triggers a subtitle search for a media item, the plugin must trigger Bazarr to perform its normal subtitle-search workflow for that exact media item.
3. Pass through the requested subtitle language and whether the request is for forced subtitles only. The current version of Bazarr should ignore these, but it is part of the subtitle system in Emby.
4. Do not return a user-selectable subtitle result list from the provider (an empty list may be required, in order to avoid error states). Instead, Bazarr should perform its own matching, scoring, and download behavior.
5. Assume Bazarr is responsible for its own provider configuration, download logic, score logic, and post-download behavior.
6. After Bazarr downloads subtitles and Emby refreshes the media item, detect whether a subtitle file is new or modified compared with the pre-search snapshot. Consider tracking pending searches within Emby's metadata files or server database, if possible, and does not violate standard practices for metadata files and Emby database usage.
7. If a new or modified subtitle file is detected, send an Emby notification to the user who triggered the search when possible.
8. Notification content must clearly identify the media item:
   - Movies: title and release year
   - Episodes: series name, season number, and episode number
9. Include rate limiting so Emby can cap how many subtitle searches per hour are sent to Bazarr. Requests beyond the cap must be queued and processed later.
10. Include a settings UI inside the Emby dashboard for configuring Bazarr access and plugin behavior.
11. Enter this plugin as a 'subtitle provider' within the Emby dashboard.
12. Provide secure API-key handling. Send the Bazarr API key in a header such as `X-API-KEY`, not in query strings or URLs.
13. Include explanatory comments throughout the code so a mid-level hobbyist developer can understand the purpose of the code, important order-of-operations choices, security decisions, and Emby/Bazarr integration constraints.
14. Check the repository description for possible additional guidance.

===============================================================================
MATCHING REQUIREMENTS
===============================================================================

Media matching between Emby metadata and Bazarr entries must support more than just Radarr/Sonarr IDs.

Implement matching logic that attempts to correlate items using, where available:

1. TMDB IDs for movies
2. TVDB IDs for series and episodes
3. Radarr IDs for movies
4. Sonarr IDs for series/episodes
5. Validate each returned match via, at a minimum:
  - title
  - year +/-1 (occasionally there are release year discrepancies)
5. As a fallback if the above methods either do not return proper matches, or return multiple matches:
  - File-path-based matching that uses cached media metadata polled from bazarr
  - The cached metadata to be polled occasionally, with a configurable TTL

Design the matching logic so ID-based matching is preferred when available, with media-file-path matching for resilience.

===============================================================================
TECHNICAL / ARCHITECTURAL REQUIREMENTS
===============================================================================

1. Base the repository structure and codebase on primarily the Emby SDK referenced later in this prompt.
2. Do not clone or mirror an older repository structure blindly. Create a clean, maintainable architecture appropriate for a new plugin repository.
3. Target `.NET Standard 2.0` / `netstandard2.0` for broad compatibility across supported .NET implementations commonly used by Emby plugin environments. If the chosen Emby SDK samples or package requirements strictly require a different target, or have been updated to newer versions of dependencies, explain these choices clearly in the README and project files.
4. Include heavy debug logging throughout the plugin so troubleshooting in real environments is easier. Log:
   - request entry/exit points
   - media matching decisions
   - ID sources used
   - queue/rate-limit behavior
   - Bazarr API request intent and response status
   - subtitle snapshot comparisons
   - notification decisions
   - recoverable failures and retry/defer behavior
5. Keep logs useful but avoid leaking secrets such as API keys.
6. Include a testable, maintainable separation of concerns for:
   - plugin entry/configuration
   - Bazarr API communication
   - media matching
   - rate limiting / queuing
   - subtitle snapshot detection
   - notifications
   - background monitoring or refresh handling
   - settings UI / configuration endpoints
7. Prefer robust, production-ready code patterns over minimal demo-only code.
8. Never transmit protected information, such as API keys, in plain text or urls. API keys and similar must always be placed in headers unless the developer explicitly is warned of and approves of edge-cases.

WORKFLOW / BRANCH / RELEASE REQUIREMENTS
===============================================================================

Use `dev` as the default branch for active development, testing, troubleshooting, and development-release validation. Treat `main` as the stable promotion branch, not the day-to-day development branch.

Implement repository automation with these rules:

1. Prefer a simple two-branch model:
   - `dev` = active integration branch for ongoing work
   - `main` = stable branch for promoted releases
2. New work should be based on `dev`, merged into `dev` for validation first, and promoted to `main` only when ready for a clearly stable release.
3. The `dev` branch must build automatically and produce a downloadable development artifact or prerelease `dll` file.
4. The `dev` output must be clearly marked as a development build, with `dev` included in the version, tag, artifact, or release identifier.
5. Each `dev` release should be automatically version-incremented.
6. The `main` branch must also build automatically and produce a full stable DLL artifact suitable for direct Emby installation.
7. Stable `main` releases must remain clearly distinguishable from `dev` prereleases/drafts.
8. Each `main` release should be automatically version-incremented.
9. A simpler acceptable variant is: automatic CI builds on all branches, automatic development prerelease publication only from `dev`, and automatic stable release publication only from `main`. If this variant is chosen, document it clearly in the README.
10. Preserve a similar level of workflow usefulness as:
   - CI build artifacts downloadable from workflow runs
   - release artifacts downloadable from GitHub Releases
   - branch-aware stable vs development release behavior
11. Document the workflow behavior clearly in the README.

IMPLEMENTATION NOTES
===============================================================================

The generated repository should embody these design expectations:

1. Emby triggers the search request.
2. The plugin records the current subtitle-file snapshot for the media item OR inserts trigger information into either the metadata files or Emby database.
3. The plugin resolves the media item against Bazarr using preferred ID matching first, then file-path fallback to cached data.
4. The plugin triggers Bazarr for the requested language / forced state.
5. The plugin returns control to Emby without presenting a manual subtitle-choice list. Consider dummy return data that indicates a search will be performed in the background.
6. After Bazarr finishes and Emby refreshes the item, the plugin detects new/modified subtitle files.
7. The plugin sends an Emby notification about success or important failure states.
8. After Bazarr successfully receives a search trigger, the search is removed from the pending searches within the plugin.

Security expectations:

1. Never place API keys in URLs, query parameters, logs, README examples that imply insecure transport, or client-visible browser history where avoidable.
2. Make such protected data in logs conform to Emby's redaction methodology enabled by "anonymize log contents" that is available when viewing a log file within the Emby dashboard.
2. Prefer headers and secure configuration handling.
3. Be explicit in code comments and README about why the secure transport pattern is used.
